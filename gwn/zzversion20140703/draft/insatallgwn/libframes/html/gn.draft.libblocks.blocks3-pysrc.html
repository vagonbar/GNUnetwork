  4  ''' 
  5  An experimental block implementation. 
  6   
  7  Each block has a single "input queue", and several "input ports". All events are received and queued in the input queue. When an event is got from the input queue, it is handed to all input ports. Ideally, only one input port understands it and accepts it; the rest ignore the event. 
  8   
  9  Input ports are objects, and so can retain state, but do not exhibit behavior. Processing of events in the input lists is done by the  
 10   
 11  The test:  an RTS - CTS - Data - ACK dialog between two blocks. 
 12   
 13  The blocks and ports: 
 14   
 15             Block1                   Block2 
 16               in_port 0 ------------> block 2 input queue 
 17  events --->  block 1 input queue <---- in_port 0 
 18               in_port 1 
 19   
 20  Block 1 receives in its input queue a sequence of events, which may be Configuration, Data, CTS, ACK or other. Port 1 accepts configuration events Conf, Reset, and discards others; processing events on port 1 can send OK events. Port 0 accepts Data, CTS, ACK events and discards others; processing events on port 0 can send RTS, Data. 
 21   
 22  Block 2 receives events from Block 1 in its input queue; port 0 accepts RTS, Data events, and discards others; processing events on port 0 can send CTS, ACK. 
 23   
 24
